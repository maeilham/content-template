# Contributing

매일함 콘텐츠 저장소의 작성 규칙입니다.

## 파일 위치와 이름

- 모든 콘텐츠는 `content/` 디렉토리 안에 둡니다.
- 파일명: `NNNN-slug.md`
  - `NNNN`: 4자리 zero-padded 숫자 (예: `0001`, `0042`). repo 안에서 unique.
  - `slug`: 소문자, 영문/숫자/하이픈만. (예: `scale-up-vs-scale-out`)
- 4자리 숫자가 **발송 순서**가 됩니다. 작은 숫자부터 발송됩니다.

## frontmatter

각 콘텐츠 파일은 YAML frontmatter로 시작합니다.

```yaml
---
title: "스케일업과 스케일아웃의 차이는?"
preview: "서버 한계를 마주했을 때 떠올리는 두 가지 선택지."
tags: [infra, scaling, aws]
source:                              # (선택) 외부 자료 인용 시
  url: "https://example.com/source"
  author: "원작자명"
---
```

### 필수 필드

| 필드 | 설명 |
|---|---|
| `title` | 메일 제목 / 사이트 표시 제목 |
| `preview` | 메일에 노출되는 1-2줄 미리보기. 호기심을 자극하는 톤이 좋습니다 |

### 선택 필드

| 필드 | 설명 |
|---|---|
| `tags` | 검색·연관 추천에 활용. 자유 배열 |
| `source.url` | 다른 자료를 인용했다면 출처 URL |
| `source.author` | 원작자 표기 |

> 외부 자료를 가져온 경우 **반드시 source를 명시**해 주세요.
> CC BY 호환 라이선스의 자료만 가져오는 것을 권장합니다.

## 본문 구조

frontmatter 다음 본문은 아래 구조를 따릅니다.

```markdown
## 질문
{질문 본문}

## 답안 예시
{짧은 답안 예시 — 정답이 아닌 한 가지 시각}

## 참고 자료
- {링크 또는 출처}
```

- 사용자는 메일에서 `preview` 만 봅니다. 전체 본문은 사이트/GitHub에서 봅니다.
- "답안 예시"는 **정답이 아니라 한 가지 시각**으로 짧게 적어 주세요.
  사용자가 직접 생각하고 Discussion에서 토론할 여지를 남기는 것이 매일함의 정신입니다.
- 답안 예시는 가능하면 200~300자 안으로 짧게 유지해 주세요.

## PR 워크플로우

1. fork 또는 새 브랜치 생성 (`feat/0042-some-topic`)
2. `content/NNNN-slug.md` 추가
3. 셀프 체크리스트 통과 후 PR 생성
4. 메인테이너 리뷰 후 머지

### 셀프 체크리스트

- [ ] 파일명이 `NNNN-slug.md` 형식인가요?
- [ ] `title`, `preview` 가 있나요?
- [ ] 본문에 `## 질문`, `## 답안 예시`, `## 참고 자료` 섹션이 있나요?
- [ ] 답안 예시가 너무 길지 않나요?
- [ ] 외부 자료 인용 시 `source` 필드를 채웠나요?

## 라이선스
